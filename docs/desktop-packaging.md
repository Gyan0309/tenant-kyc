# Desktop Packaging

Tenant Manager ships as a Windows desktop app: an Electron shell that runs the
Next.js server locally and displays it in a native window. The app still reads
and writes to Azure Storage — the desktop shell just replaces the browser +
`next start`.

## How it works

```text
Tenant Manager.exe (Electron)
  main process (electron/main.js)
    reads per-install config.json  ──▶  Azure connection strings + AUTH secret
    spawns the Next.js standalone server on a random 127.0.0.1 port
    opens a BrowserWindow pointed at that server
        │
        ▼
  Next.js standalone server (.next/standalone/server.js)
    App Router pages + API routes  ──▶  Azure Table + Blob Storage
```

- `next.config.ts` sets `output: "standalone"`, so `next build` emits a
  self-contained server at `.next/standalone` that runs without a full
  `node_modules` install.
- The Electron **main process** (`electron/main.js`) launches that server as a
  child process using Electron's bundled Node (`ELECTRON_RUN_AS_NODE`), waits
  for it to accept connections, then loads it in a window.
- The renderer only ever talks to `http://127.0.0.1:<port>` (the local Next
  server). `contextIsolation` is on and `nodeIntegration` is off.

## Per-install configuration (the multi-client seam)

**Azure connection strings are never baked into the app bundle.** On first
launch the app creates a config file in the OS user-data directory:

```text
Windows: %APPDATA%\Tenant Manager\config.json
```

```json
{
  "AZURE_TABLE_CONNECTION_STRING": "",
  "AZURE_STORAGE_CONNECTION_STRING": "",
  "AZURE_BLOB_CONTAINER_DOCS": "tenant-documents",
  "AZURE_BLOB_CONTAINER_ASSETS": "property-assets",
  "AUTH_SECRET": "<auto-generated on first run>"
}
```

If the two `AZURE_*` values are empty, the app opens this file and quits with a
message asking the operator to fill them in.

This file is the seam that makes **"one shared Azure account now, a dedicated
account per client later"** a config edit rather than a rebuild:

- **Now:** every install gets the *same* connection strings → one shared Azure
  Storage account. Client data is isolated only by the app-layer `ownerId`
  check (see `lib/auth/session.ts` and the per-row `ownerId` comparison in the
  API routes).
- **Later ("when they get serious"):** give a client's install its *own*
  storage account connection strings in this file and run `npm run bootstrap`
  against that account. Same binary, different config, real physical isolation.
  No code change — `lib/azure/tables.ts` and `lib/azure/blobs.ts` already read
  the strings from the environment.

> Security note: while multiple clients share one account with the string sitting
> on client machines, isolation is app-layer only. That is an acceptable early
> tradeoff, but moving each client to their own storage account is the thing to
> do before this handles sensitive data at scale.

## Build commands

```bash
npm run build:desktop   # next build (standalone) + copy public/ and .next/static into standalone
npm run desktop         # run the Electron shell against the local standalone build
npm run dist            # build:desktop + package a Windows NSIS installer into dist-desktop/
npm run dist:dir        # build:desktop + unpacked app only (dist-desktop/win-unpacked/), no installer
```

Artifacts land in `dist-desktop/`:

- `Tenant Manager Setup <version>.exe` — the installer (NSIS, user-selectable
  install dir, per-user).
- `win-unpacked/Tenant Manager.exe` — the unpacked app for quick testing.

The standalone server is packaged as an unpacked `extraResources` folder
(`resources/standalone/`) so it can be executed from disk at runtime.

## Building the installer on Windows (winCodeSign symlink gotcha)

`electron-builder` downloads a `winCodeSign` bundle that contains macOS symlinks.
Extracting them requires the Windows privilege to create symbolic links, which
non-admin accounts lack unless **Developer Mode** is enabled. Symptom:

```text
ERROR: Cannot create symbolic link : A required privilege is not held by the client. : ...libcrypto.dylib
```

Pick one:

- Enable **Windows Developer Mode** (Settings → System → For developers), or run
  the build from an elevated (admin) terminal — either grants the symlink
  privilege, and `npm run dist` works directly.
- Or, without changing OS settings, pre-seed the cache with a symlink-free
  extraction so the download step is skipped. The macOS files are irrelevant to
  a Windows build:

  ```powershell
  $cache = "$env:LOCALAPPDATA\electron-builder\Cache\winCodeSign"
  $za = ".\node_modules\7zip-bin\win\x64\7za.exe"
  # extract any downloaded winCodeSign-*.7z there with -snld (skips symlinks),
  # copy it to "$cache\winCodeSign-2.6.0", and delete its "darwin" subfolder.
  ```

Code signing is skipped by default (no certificate). Distributing unsigned means
users see a SmartScreen warning; add a code-signing certificate later via
`electron-builder`'s `win.certificateFile` / `CSC_*` env vars.
