# Deployment Guide — build, ship, install, configure

This is the end-to-end guide for turning the source into a Windows app a client
can install, and pointing that install at the right Azure Storage account.

Product name (shown to users): **Tenant Manager**. Vendor: SAGA (in
`package.json` only, not shown in the UI).

---

## 0. One-time: what you need on the build machine

- Node.js 20+
- The repo cloned, with dependencies installed:
  ```bash
  npm install --legacy-peer-deps
  ```
- An Azure Storage account (or one per client — see §5). You need its
  **connection string** (Azure Portal → Storage account → Access keys →
  Connection string).

You do **not** need Azure to *build* the app — only to *run* it.

---

## 1. Build the Windows installer (.exe)

```bash
npm run dist
```

This runs `next build` (standalone), copies assets, generates the app icon, and
packages everything with electron-builder. Output lands in `dist-desktop/`:

| File | What it is |
|---|---|
| `Tenant Manager Setup <version>.exe` | **The installer you send to clients.** |
| `win-unpacked/Tenant Manager.exe` | Unpacked app for your own quick testing. |
| `latest.yml`, `*.blockmap` | Auto-update metadata (only needed if you add updates later). |

### If the build fails with a "symbolic link" error

electron-builder downloads a code-signing bundle containing macOS symlinks that
Windows can't extract without Developer Mode/admin. Fix (pick one):

- Turn on **Windows Developer Mode** (Settings → System → For developers), or run
  the build from an **admin** terminal, then `npm run dist` again; **or**
- Pre-seed the cache without symlinks — see `docs/desktop-packaging.md` →
  "winCodeSign symlink gotcha".

> The installer is **unsigned**, so Windows SmartScreen will warn on first run
> ("More info → Run anyway"). To remove the warning, add a code-signing
> certificate later (`win.certificateFile` / `CSC_*` env vars).

---

## 2. Send the installer to the client

Send **only** `Tenant Manager Setup <version>.exe` (e.g. via download link).
Nothing else is required — the Next.js server and all assets are bundled inside.

Do **not** put any Azure connection string inside the installer. Configuration
happens per-install, on the client machine (§4).

---

## 3. Client installs the app

The client runs `Tenant Manager Setup <version>.exe`:

1. If SmartScreen warns, "More info → Run anyway".
2. Choose an install folder (per-user install; no admin needed).
3. Finish → a **Tenant Manager** shortcut is created.

On first launch, the app has no storage configured yet, so it will:

- create a config file, then
- pop a message ("configuration needed") and open that file, then quit.

This is expected — go to §4.

---

## 4. Configure the connection string — the easy way (recommended)

Don't make the client edit files. Use the ready-made configurator in
[`client-setup/`](../client-setup/README.md):

1. **You:** open `client-setup/Configure-TenantManager.ps1`, paste the client's
   two Azure connection strings near the top, save.
2. **You:** send the client both `Configure Tenant Manager.cmd` and
   `Configure-TenantManager.ps1` (zip them together).
3. **Client:** double-clicks **`Configure Tenant Manager.cmd`**.

The script writes the config to the correct location, generates the security
secret, and launches the app — the client never touches JSON or an Azure key.
Re-running it is safe (keeps the existing login secret).

The manual method below is only if you prefer to hand over a ready file yourself.

## 4b. Configure the connection string manually (where + how)

The app reads its settings from a per-install JSON file in the Windows user
profile:

```
%APPDATA%\Tenant Manager\config.json
```

(Paste that path into the Explorer address bar, or use the file the app opened
for you on first launch.)

It looks like this:

```json
{
  "AZURE_TABLE_CONNECTION_STRING": "",
  "AZURE_STORAGE_CONNECTION_STRING": "",
  "AZURE_BLOB_CONTAINER_DOCS": "tenant-documents",
  "AZURE_BLOB_CONTAINER_ASSETS": "property-assets",
  "AUTH_SECRET": "<auto-generated — leave as is>"
}
```

Fill in the **two** Azure connection strings, save, and reopen the app:

| Field | Where it comes from |
|---|---|
| `AZURE_TABLE_CONNECTION_STRING` | Azure Portal → Storage account → Access keys → **Connection string**. |
| `AZURE_STORAGE_CONNECTION_STRING` | Same connection string (Table + Blob live in one account). |
| `AZURE_BLOB_CONTAINER_DOCS` / `_ASSETS` | Leave as default unless you renamed containers. |
| `AUTH_SECRET` | Auto-generated on first run. Don't change it (changing it signs everyone out). |

You (the vendor) normally do this once per client and hand over a ready
`config.json`, so the client never touches Azure keys.

> **Tables and containers are created automatically.** On startup the app
> provisions the `Owners/Properties/Rooms/Persons/Documents/ConsentLogs` tables
> and the blob containers if they don't exist — no manual `npm run bootstrap` on
> the client. (You only run `npm run bootstrap` if you want to pre-create them
> against an account yourself.)

After saving, reopen **Tenant Manager** → it starts, and the client registers an
owner account and begins adding properties.

---

## 5. One account now, per-client accounts later

The connection string is **config, not code**, so scaling isolation is a config
edit, not a rebuild:

- **Now (cheap):** every client's `config.json` points at the **same** Azure
  account. Data is separated only by the in-app owner check.
- **Later (proper isolation):** give a client a `config.json` pointing at **their
  own** Azure Storage account. Same installer, different config. The app
  auto-creates that account's tables/containers on first launch.

See `docs/desktop-packaging.md` for the security trade-off and the architecture
behind this.

---

## 6. Shipping a new version

1. Bump `version` in `package.json`.
2. `npm run dist`.
3. Send the new `Tenant Manager Setup <version>.exe`. Installing over an existing
   install keeps the client's `config.json` and their data (data lives in Azure,
   not in the app).

---

## Quick reference

```bash
npm install --legacy-peer-deps   # once, on the build machine
npm run dist                     # build the installer -> dist-desktop/
npm run desktop                  # run the shell locally against a local build
npm run gen:icon                 # regenerate the app icon after editing it
```

Client config file: `%APPDATA%\Tenant Manager\config.json`
